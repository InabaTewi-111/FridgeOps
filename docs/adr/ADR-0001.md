# ADR-0001: API は REST API（API Gateway）で行く。MVP の Contract も固定する

- Status: Accepted
- Date: 2026-02-05

## 何を決める？
MVP の API を REST / HTTP のどっちで作るか。
あと、フロントとインフラが迷子にならないように、最初に Contract を固定する。

## 結論
API Gateway は REST API を採用する（HTTP API は今回は見送る）。

## そうした理由（メモ）
- 「コストブレーキ」や「運用の統制」を語れる形にしたい  
  → Usage Plan / API Key / Throttling の話につなげやすいのが REST 側
- 将来の拡張（認証・検証・段階運用）を考えた時に、説明できる引き出しが多い
- HTTP API は軽くて安いけど、今回は “安さ” より “コントロール性と説明可能性” を優先する

## 影響
- REST の方が多少重い/高い可能性あり（ただし MVP の範囲なら許容）
- 以降の実装は下の Contract（v1）前提で進める（勝手に変えない）

## MVP API Contract（v1 固定）
Base: /api

### GET /api/items
一覧を返す。
Response 200:
